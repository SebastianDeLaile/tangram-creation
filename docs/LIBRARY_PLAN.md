# Library plan: sourcing 100+ tangram figures

Goal: a library of at least 100 validated tangram configurations, organized
by category. Figures are **sourced from existing, real tangram puzzles**
(books, open-source projects, puzzle datasets) rather than generated by
randomly scattering pieces — random placement produces legal-but-meaningless
shapes, not recognizable figures.

## Categories

| Category | Description | Count so far |
|---|---|---|
| `geometric` | The 13 convex tangram shapes (mathematically complete set), plus the original square | 10 |
| `animals` | Birds, cats, rabbits, fish, horses, etc. | 26 |
| `people` | Running, sitting, dancing, kneeling figures | 0 |
| `objects` | Houses, boats, candles, arrows, mountains, bridges | 19 |
| `letters` | Tangram forms of letters/numbers | 6 |
| `abstract` | Structurally valid configurations with no specific silhouette (useful for testing, not display) | 17 |

`examples/index.json` is the source of truth for category assignment — every
file in `examples/` must be listed there with a `category`, and
`tests/test_validate.py::test_index_lists_exactly_the_files_present` enforces
that the manifest and the directory never drift apart.

## Sourcing strategy

In order of preference:

1. **Structured piece-placement data** — sources that publish each piece's
   anchor/orientation/flip (or equivalent), not just a silhouette outline.
   This is directly convertible and the conversion is mechanically checkable
   (see Validation below). Example: **Wiebke/TangramGenerator**
   (github.com/Wiebke/TangramGenerator, MIT licensed) ships
   `Code/exampleTangrams.js` with named figures (square, swan, cat, bird,
   mountain, arrow, two letters) plus six more in raw JSON. Its coordinate
   system and canonical piece directions are identical to this project's
   (`pieces.CANONICAL_DIRECTIONS`), confirmed by parsing `Code/directions.js`
   — so figures translate over with zero coordinate adjustment. This is
   where all 14 current figures (besides the constructed square) come from.

2. **Silhouette-only datasets** — sources with just the outline shape, no
   solved piece placement. Found `twpride/tangram`
   (github.com/twpride/tangram), which ships 179 silhouette outlines in
   `problemsData.js` (raw polygon point lists), but no piece solutions —
   solving each one means writing a tangram solver (search over piece
   placements that exactly tile a given outline). Useful as a source of
   *category ideas and target shapes* later, not for direct import now.

3. **Public-domain books** — Sam Loyd's "The 8th Book of Tan" (1903, public
   domain) has ~700 silhouettes, but only as images; turning those into piece
   coordinates requires manual digitization or computer vision. Treat as a
   future, higher-effort source once cheaper sources are exhausted.

4. **Computed sets** — the 13 convex tangrams are a proven-complete
   mathematical set with published coordinates; derive these directly rather
   than searching for a dataset.

Repos checked and rejected for now (no usable piece-placement or outline
data): `aniamika/Tangram`, `shgalus/tangram`.

## Conversion pipeline

`scripts/import_tangram_generator.py` is the reusable importer for source 1
above: it regex-parses the `Point`/`Tan`/`Tangram` JS constructor calls (and
the raw embedded JSON `interesting*` figures) out of `exampleTangrams.js`,
maps each source `tanType` (0-5, where 5 is the flipped parallelogram) to
this project's `PieceType` + `flipped`, and builds `Tangram` objects directly
— no coordinate transform needed, since the source and this project share
the same anchor/direction/orientation convention. Run it standalone to
validate a batch before committing anything:

```bash
curl -s https://raw.githubusercontent.com/Wiebke/TangramGenerator/master/Code/exampleTangrams.js \
    -o /tmp/exampleTangrams.js
.venv/bin/python3 scripts/import_tangram_generator.py /tmp/exampleTangrams.js
```

It prints `OK` or the specific overlap/count problems per figure; nothing is
written to `examples/` until a figure is confirmed valid and given a
deliberate name, description, and category in `examples/index.json`.

The same shape of script (parse source's piece data -> map to `PieceType` ->
build `Tangram` -> `validate()` -> name + categorize -> `save_tangram`) is the
template for any future structured source.

## Validation gate

Every figure added to `examples/` must pass `tangram.validate.validate()`
before being committed:

- exactly 7 pieces, correct counts/ids
- no pairwise overlap beyond `OVERLAP_TOLERANCE`
- all pieces connected into a single group (`is_connected`) — pieces may
  share a full edge *or* touch at a single pinch point (both are legitimate
  in real tangram art, e.g. a wingtip touching a body at one vertex), but
  there must be no genuine gap between groups

This is enforced by `tests/test_validate.py`, which parametrizes over every
entry in `examples/index.json` — so a bad figure fails CI, not just a
one-off check at import time.

**Incident:** two imported figures (`letter_t`, `letter_g`, from
TangramGenerator's less-exercised `bigT`/`bigG` demo entries) turned out to
have pieces placed several units apart with no shared vertex at all — a
real defect in the *source* data, faithfully transcribed by the importer.
`validate()` didn't catch it at the time because it only checked overlap,
not connectivity; a visual render of one suspect figure (`letter_t` looked
like two disconnected clusters) caught it, and a connectivity check
(`is_connected`, vertex/edge-touching graph over all 7 pieces) was added to
`validate()` to catch this class of bug automatically going forward. Both
figures were removed. Lesson: spot-check new figures with `render.save_svg`
before trusting them, even when the source is otherwise reliable — a
parser faithfully transcribing bad source data is not the same as a correct
figure, and our own tooling (here, a freshly-written boundary/connectivity
checker) needs the same skepticism (its first version had a false-positive
bug on legitimate pinch-point figures, caught the same way: render and look).

## Status and next steps

**216 figures committed** (target of 100+ exceeded). The big jump from ~103 came
from two solver upgrades in `scripts/import_nevit_svg.py`:

- A **backtracking placement solver** (`_solve_by_backtracking`) that, since piece
  type/orientation/flip are already known from the SVG fit, searches only for each
  piece's exact ℤ[√2] anchor. Key tricks: infer the figure's shared b-residue mod 6
  (Z[√2] is dense, so this prunes spurious lattice approximations), order the search
  by float-adjacency with incremental connectivity pruning, and use point-to-edge
  distance so T-junctions register as touching. This recovered ~75 figures that the
  greedy BFS/edge methods left disconnected.
- **Per-figure `allow_disconnected`** support: some real figures are drawn as
  several intentionally-separated parts (a candle flame above the body, a saddle
  above a horse). `_solve_faithful` snaps every piece to the grid with no-overlap
  but no connectivity requirement, and `validate()` skips the connectivity check for
  figures that set the flag (38 such figures). The flag round-trips through the JSON
  schema on both the Python and TS sides.

The ~38 still-unimported Nevit files have non-standard piece angles (fit RMSE too
high) and would need a different fitting approach.
